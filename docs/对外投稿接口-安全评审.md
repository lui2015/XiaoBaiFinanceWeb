# 对外投稿接口 安全评审报告

> 评审对象：`POST /api/open/articles`（对外开放投稿）及其关联链路
> 关联文件：`src/app/api/open/articles/route.ts`、`src/app/api/open/docs/route.ts`、`src/lib/sanitize.ts`、`src/components/ArticleHtml.tsx`、`src/lib/utils.ts`
> 评审日期：2026-07-17
> 状态：**待评审**（暂未修复，等待团队确认修复方案）

---

## 1. 背景

对外投稿接口经历了两步演进：

1. 移除原有的 API Key 鉴权，改为公开匿名调用；
2. 投稿由"待审核"改为**直接发布**（`status=1`），无需人工审核即对外可见。

即当前状态为 **无鉴权 + 直接发布**。本报告聚焦这一组合带来的风险，供团队评审是否可接受、以及确定修复方案。

---

## 2. 接口概览

| 接口 | 方法 | 鉴权 | 行为 |
|---|---|---|---|
| `/api/open/articles` | POST | **无** | 投稿内容净化后**直接发布**（status=1），立即公开可见，并同步搜索索引 |
| `/api/open/docs` | GET | 无 | 返回投稿说明文档（Markdown 纯文本） |

### 现有防护措施
- 内容强制净化（防 XSS）；
- 按来源 IP 限流：`fixedWindow('open:submit:ip:${ip}', 20, 60)`（每 IP 60 秒内 20 次）；
- 分类必须存在且启用；
- slug 冲突自动追加随机后缀；
- 审计留痕（`operationLog`，记录来源 IP / UA）；
- 渲染侧使用 sandbox iframe 隔离脚本与站点 Cookie/DOM。

### 值得肯定的设计
- 内容净化 + 渲染侧 sandbox iframe（无 `allow-same-origin`）形成**纵深防御**，脚本入库即被剔除；
- zod 参数校验、分类有效性校验、slug 唯一化、审计留痕、索引同步失败降级不阻断主流程；
- `docs` 接口读取固定路径、无用户输入拼接，**无路径穿越（Path Traversal）风险**。

## 3. 问题清单

### 🔴 P0-1：限流可被 `X-Forwarded-For` 伪造绕过（叠加无鉴权 = 可无限刷库）

**位置：** `src/lib/utils.ts`

```ts
export function getClientIp(req: NextRequest): string {
  const xff = req.headers.get('x-forwarded-for');
  if (xff) return xff.split(',')[0].trim();   // 取 XFF 第一个值——完全由客户端控制
  return req.headers.get('x-real-ip') || '0.0.0.0';
}
```

**问题描述：**
`getClientIp` 直接取 `X-Forwarded-For` 的**第一个值**，而该值完全由客户端控制，可任意伪造。限流 key 为 `open:submit:ip:${ip}`，攻击者每次请求携带一个随机 `X-Forwarded-For`，key 每次都不同，限流形同虚设。

**为何严重：**
在无鉴权场景下，IP 限流是**唯一的防滥用手段**，一旦被架空，等于零防护。

**影响：**
- 可批量灌入垃圾 / 广告 / 违规文章（单篇正文上限 200KB）；
- 内容**直接对外发布**并同步进搜索索引，污染站点内容与 SEO；
- 被刷后清理与处置成本高。

**复现思路：**
```bash
# 每次带不同的伪造 IP，即可绕过 60s/20 次的限制
for i in $(seq 1 100); do
  curl -s -X POST http://<host>/api/open/articles \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 1.2.3.$((RANDOM % 255))" \
    -d '{"title":"spam","categorySlug":"basics","contentHtml":"<p>spam</p>"}'
done
```

**修复建议（二选一或叠加）：**
- 生产环境只信任可信反向代理（如 nginx）注入的 XFF，取**最后一跳**而非第一个值；或直接改用 `x-real-ip`。开发环境用 socket 地址兜底；
- 恢复轻量准入门槛：API Key（env-only）或将投稿改回"待审核"（`status=0`），经人工 / 规则 / AI 审核后再发布。

### 🟠 P1-1：无鉴权直接发布的产品与内容风险

**位置：** `src/app/api/open/articles/route.ts`

**问题描述：**
任何匿名请求都能让内容**立即上线**首页 / 分类页 / 搜索结果。即便限流问题修复，"匿名即可发布对外可见内容"本身仍是较高的产品与合规风险（涉黄涉政、广告、诽谤等内容将由平台承担责任）。

**影响：**
- 内容合规责任落到平台侧；
- 缺少发布前闸门，一旦被利用，负面内容即时可见；
- 目前虽有 `operationLog` 记录来源 IP，但缺少便捷的一键下架 / 批量清理入口，被刷后处置成本高。

**修复建议：**
- 至少落地"先入库待审核"（`status=0`），由人工 / 规则 / AI 审核后再发布；
- 提供按来源 IP / 时间段批量下架、批量删除的管理能力。

---

### 🟠 P1-2：sandbox 允许弹窗逃逸 + 富 HTML 未限制 iframe / 外链（钓鱼 / 引流）

**位置：** `src/components/ArticleHtml.tsx`、`src/lib/sanitize.ts`

```tsx
sandbox="allow-scripts allow-popups allow-popups-to-escape-sandbox"
```

**问题描述：**
1. 渲染侧 sandbox 含 `allow-popups allow-popups-to-escape-sandbox`，配合注入的 `<base target="_blank">`，投稿内容里的链接 / 表单可打开新窗口"逃出"沙盒跳转到外站，可用于**钓鱼**。
2. `sourceType=0`（默认）走的是 `sanitizeRichHtml`，该函数**非常宽松**——仅剔除 `<script>` / `on*` 事件 / 危险协议 URL，**不限制 iframe 的 host**（不同于严格白名单的 `sanitizeHtmlContent`，后者只放行 B 站 / 腾讯视频 / YouTube）。因此可嵌入任意外站 iframe 做引流 / 点击劫持，外链 `<img>` 也可用于用户追踪。

**影响：**
- 钓鱼跳转、外站引流、点击劫持；
- 外链资源可追踪访问者。

**修复建议：**
- 匿名投稿的 `sourceType=0` 强制改用**严格白名单**净化（`markdownToSanitizedHtml` / `sanitizeHtmlContent`，带 iframe host 白名单），把宽松的 `sanitizeRichHtml` 仅保留给可信的后台 / 前台管理上传路径；
- 考虑去掉 `allow-popups-to-escape-sandbox`。

## 5. 风险汇总与建议优先级

| 级别 | 编号 | 问题 | 建议处理时机 |
|---|---|---|---|
| 🔴 P0 | P0-1 | 限流被 XFF 伪造绕过（无鉴权下等于零防护） | **立即修复** |
| 🟠 P1 | P1-1 | 无鉴权直接发布的产品 / 合规风险 | 尽快确定策略 |
| 🟠 P1 | P1-2 | sandbox 弹窗逃逸 + 富 HTML 未限制 iframe / 外链 | 尽快修复 |
| 🟡 L | L-1 | 子分类归属未校验 | 排期修复 |
| 🟡 L | L-2 | 标签串行写入 | 可选优化 |

### 推荐的最小改动（改动小、收益大）
1. 修 `getClientIp`：生产信任反代最后一跳 / 改用 `x-real-ip`，堵住限流绕过（对应 P0-1）；
2. 匿名投稿 `sourceType=0` 改用严格白名单净化 + iframe host 白名单（对应 P1-2）；
3. 团队评审决定是否恢复"待审核"或加 API Key 准入（对应 P1-1）。
